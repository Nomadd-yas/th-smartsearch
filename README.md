## Возможности

- **Поиск СТЕ** — многоиндексный поиск с морфологией, биграммами, фаззи-матчингом и RRF-ранжированием
- **Взаимозаменяемые СТЕ** — автоматический подбор коммерчески эквивалентных позиций по характеристикам
- **Расчёт НМЦК** — метод сопоставимых рыночных цен: IQR-фильтрация, оптимальное скользящее окно
- **Workspace-сессии** — сохранение состояния поиска в Redis с ручными корректировками контрактов
- **Корзина** — хранение позиций закупки с рассчитанной НМЦК и текстовым обоснованием
- **Отчёты** — генерация обоснования НМЦК в формате DOCX с историей документов
- **JWT-аутентификация** — access + opaque refresh токены, автоматическое создание пользователя `admin` при первом запуске

## Поисковый движок

Движок построен поверх цепочки наследования `v4 → v5 → v6` и объединяет несколько независимых индексов.

### Индексы и их роли

| Индекс | Вес | Описание |
| --- | --- | --- |
| **Символьные n-граммы** (3, 4) | 0.20 | Устойчивость к опечаткам и сокращениям |
| **Инвертированный индекс** (леммы) | 0.17 | BM25-скоринг по нормализованным словам |
| **Биграммный индекс** | 0.28 | Фразовые пары лемм — ключевой сигнал точного совпадения |
| **Фаззи-скоринг** (rapidfuzz) | 0.17 | `WRatio` + `partial_ratio` — нечёткое сходство строк |
| **Индекс наименований** | 0.12 | BM25 по полному нормализованному названию |
| **Форматные токены** | 0.04 | Точное совпадение кодов (`a4`, `ip65`, `220v`) |
| **Атрибутный индекс** | 0.02 | Числовые и буквенно-цифровые коды (`5w40`, `80г`) |

### Схема ранжирования

```text
Запрос
  │
  ├─ Токенизация + лемматизация (pymorphy3)
  ├─ Извлечение форматных / атрибутных токенов
  │
  ├─ [char-индекс]    →  top-150 кандидатов
  ├─ [word-индекс]    →  top-150 кандидатов   ── RRF слияние → top-150
  ├─ [bigram-индекс]  →  top-150 кандидатов        score = Σ 1/(k + rank + 1)
  └─ [name-индекс]    →  top-150 кандидатов
          │
          ▼
  Финальный скоринг (взвешенная сумма по 7 индексам)
          │
          ▼
  Coverage-фильтр (мягкий порог покрытия слов запроса >= 0.62)
          │
          ▼
  top-K результатов (default: 20)
```

### Coverage-фильтрация

Для запросов из 2+ смысловых слов документ штрафуется, если покрывает меньше **62%** слов запроса:

```text
final_score = raw_score x coverage^0.7
```

При отсутствии результатов порог автоматически снижается: `0.62 → 0.40 → 0`.

### Взаимозаменяемые СТЕ (v6)

Characteristic Similarity Index хранит `(нормализованный_ключ, нормализованное_значение) → {ste_id, ...}`.

Три уровня приоритета характеристик:

| Tier | Вес | Примеры ключей |
| --- | --- | --- |
| **1 — функциональное тождество** | 0.7–1.0 | МНН, дозировка, тип, форма выпуска, назначение |
| **2 — состав и ресурс** | 0.4–0.6 | Материал, состав, ресурс, объём |
| **3 — технические параметры** | 0.15–0.3 | Размер, мощность, формат, цвет |

Условие совпадения: наличие ≥1 совпадения по Tier-1.
Итоговый скор: `Σ(веса совпавших ключей) / Σ(веса всех ключей источника)`.

## Алгоритм НМЦК

Реализован в `utils/nmck.py` по методу сопоставимых рыночных цен.

<details>
<summary><b>Подробное описание шагов</b></summary>

1. **Фильтрация** (последовательно):
   - Регион поставщика — точное совпадение (без учёта регистра)
   - Единица измерения — поддерживает мультизначения `л;см³`
   - Ставка НДС — точное совпадение

2. **Временной диапазон**:
   - Явный диапазон (`date_from` / `date_to`) — если передан
   - Иначе — последние **6 месяцев** от даты последнего контракта

3. **Удаление выбросов (IQR)**:

   ```text
   lower = Q1 − 1.5 × IQR
   upper = Q3 + 1.5 × IQR
   ```

4. **Ручные корректировки**:
   - `force_include` — вернуть выброс в расчёт
   - `force_exclude` — исключить учтённый контракт

5. **Оптимальное скользящее окно** (30–180 дней, шаг 5):
   - Выбирается **максимальное** окно с CV ≤ 0.33
   - При отсутствии такого — окно с **минимальным CV**

6. **НМЦК = медиана** цен в выбранном окне

</details>

Каждый контракт получает статус: `used` / `outlier` / `force_included` / `force_excluded`.

## Архитектура

```text
├── app/
│   ├── main.py              — FastAPI-приложение, lifespan, подключение роутеров
│   ├── dependencies.py      — JWT-верификация, инъекция клиента
│   ├── routers/
│   │   ├── auth.py          — Аутентификация (login, refresh, logout)
│   │   ├── search.py        — Поиск СТЕ, расчёт НМЦК, детали позиций
│   │   ├── workspace.py     — Stateful-сессии поиска (Redis)
│   │   ├── cart.py          — Корзина позиций закупки
│   │   └── report.py        — Генерация DOCX, история документов
│   └── schemas/             — Pydantic-модели запросов и ответов
│
├── utils/
│   ├── search_engine.py     — Базовый движок: инвертированный индекс, BM25, n-граммы
│   ├── search_engine_v5.py  — v5: биграммы, атрибутный индекс, RRF-ранжирование
│   ├── search_engine_v6.py  — v6: v5 + индекс схожести характеристик СТЕ
│   ├── client.py            — Обёртка над движком: поиск + загрузка контрактов
│   ├── nmck.py              — Алгоритм расчёта НМЦК
│   ├── workspace.py         — CRUD workspace-сессий в Redis
│   ├── auth.py              — Токены, пароли, управление пользователями (SQLite)
│   ├── cart.py              — CRUD корзины (SQLite, per-user)
│   ├── history.py           — История сгенерированных документов
│   ├── justification.py     — Формирование текста обоснования НМЦК
│   ├── docx_report.py       — Генерация DOCX
│   ├── contracts_by_ste.py  — Загрузка контрактов по ste_id из SQLite
│   ├── ste_details.py       — Детали СТЕ с разбором характеристик
│   ├── db.py                — Путь к БД
│   └── migrate_to_db.py     — Миграция XLSX → SQLite
│
├── config.py                — Настройки из .env (Pydantic BaseSettings)
└── run.py                   — Запуск uvicorn с CLI-аргументами
```

## Быстрый старт

### Требования

- Python 3.13+
- Redis (для workspace-сессий)

### Установка

```bash
git clone https://github.com/Nomadd-yas/th-smartsearch.git
cd th-smartsearch

python -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

### Настройка окружения

```bash
cp .env.example .env
# при необходимости отредактировать .env
```

### Подготовка данных

Поместите XLSX-файлы в папку `data/`:

```text
data/
├── TenderHack_СТЕ_*.xlsx
└── TenderHack_Контракты_*.xlsx
```

Запустить миграцию в SQLite (однократно):

```bash
python -m utils.migrate_to_db
```

> При первом поисковом запросе движок автоматически построит индекс и сохранит кэш `data/engine_cache_v6.pkl`. Повторный запуск загружает кэш за секунды.

### Запуск

```bash
# По умолчанию (настройки из .env)
python run.py

# Кастомный хост/порт
python run.py --host 0.0.0.0 --port 8080

# Режим разработки с авто-перезагрузкой
python run.py --reload
```

При старте автоматически создаётся пользователь **`admin` / `admin123`**, если база пуста.

Swagger UI: [`http://localhost:8000/docs`](http://localhost:8000/docs)
